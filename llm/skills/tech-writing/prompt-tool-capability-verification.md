# Prompt：工具能力核实

## 用途
在文章中声称某工具「支持 X 能力」之前，先核实实际能力边界，避免误导读者。

---

## 核实流程

### Step 1: 明确待验证的能力声明
例：「Comet 和 FlowithOS 可以自动解析内核补丁系列的跨 lore 依赖链」

### Step 2: 拆解能力要求
- 最小能力：能打开 URL 并读取内容？
- 专项能力：有该领域的专用索引或集成？
- 自动化能力：无需人工干预能完成全流程？

### Step 3: 独立核实

对每个工具，搜索：
- 官方文档中是否明确支持该能力
- 社区讨论/演示中是否有实证
- 工具类型（通用 agent vs 专项工具）是否匹配需求

### Step 4: 输出标准结论

```
工具名：[通用类型描述]
实际能力：[能做到什么]
能力边界：[做不到什么，原因是什么]
替代方案：[如果需要该能力，用什么工具]
文章描述建议：[如何在文章中准确表述]
```

---

## 本次案例

**待验证声明**：Comet / FlowithOS 能解决内核补丁跨系列依赖链自动拼接

| 工具 | 类型 | 实际能力 | 能力边界 | 结论 |
|---|---|---|---|---|
| Comet | 通用 AI 浏览器代理 | 可打开 lore 页面并读取内容 | 无 lore 专用索引，无法自动识别 `depends on <Message-ID>` 并跳跃拉取 | 不满足，需人工操作 |
| FlowithOS | 通用多 agent 工作流平台 | 可配置 agent 访问 lore 页面 | 无 lore 专项集成，需自建工作流，非开箱即用 | 不满足，需二次开发 |
| b4 dig | 专项 CLI | 自动发现 thread、关联邮件聚合 | 最接近的专项工具 | 推荐 |

**文章描述（修改前）**：「没有工具能自动跨 lore 跳跃拼接」
**文章描述（修改后）**：「Comet / FlowithOS 等通用 agent 不具备专项索引，无法开箱即用；`b4 dig` 是目前最接近的专项工具」

---

## 适用原则

- 通用工具（浏览器 agent、工作流平台）通常无法开箱即用替代专项工具
- 「理论上可以」≠「能用」——需要人工配置的不算开箱即用
- 核实结论写入文章时，说清楚「能做什么」和「做不到什么」
