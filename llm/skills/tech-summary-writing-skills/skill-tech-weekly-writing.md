# Skill：技术周报写作

## 适用场景
撰写底软/硬件方向的技术周报，涵盖内核补丁、芯片跑分、行业动态等主题。

---

## 文章结构规范

### 章节顺序原则
- 章节顺序应体现叙事逻辑（如：安全→基础软件→算力规模→芯片架构→制造工艺→RISC-V生态）
- 过渡句放在章节**标题之后**，作为引导段落，不放在标题之前

### 章节内部结构
```
## N、章节标题

过渡句（承接上一节，引入本节主题）

### 子节标题

正文内容
```

### 必备节
- **小结**：末尾加表格，归纳「本期核心信息一览」（主题 / 关键事实 / Take Away）
- **参考文献**：所有脚注按正文首次出现顺序编号，[^1]–[^N] 连续

---

## 内容规范

### 漏洞/安全话题
- 表格格式：CVE / CVSS / 发现方式 / 状态 / PoC 链接
- KEV 需解释（Known Exploited Vulnerabilities，CISA 已知在野利用漏洞目录）
- FCEB 需解释（Federal Civilian Executive Branch，美国联邦文职行政部门机构）
- 英文引文必须附中文译文

### 内核补丁话题
- 补丁状态区分：「正在 review」vs「完成基本补丁（已合入）」
- 依赖链用 ASCII 树展示：
  ```
  KHO core
      └── Extended scratch ← 解决 XX 问题
              └── HugeTLB preservation
  ```
- 每个补丁注明：作者、版本（RFC v1/v2）、补丁数量、提交时间、lore URL

### 跑分/性能数据
- 参见《SPEC 跑分表格 Skill》
- 所有数字必须有来源，区分实测/厂商指标/推算/估算
- SPEC2017 → SPEC2006 换算系数固定 **×9**，并备注

### 开篇
- 场景锚点开头：「这两周我在看……」「这期从一个问题开始」
- 不用「范式转移」「颠覆」「豪赌」等大词

---

## 语言风格规范

### 禁忌句式
- 「不是……而是……」→ 直接陈述
- 「这不是巧合」→「这是个新信号」
- 「正是这种……的体现」→ 删除或改为事实陈述

### 情绪表达用古文
| 场景 | 推荐用法 |
|---|---|
| 历史漏洞被 AI 发现 | 积年之患，一朝显形 |
| 原来有效的假设失效 | 刻舟求剑 |
| 两方此消彼长 | 此消彼长 |

### 英文引文格式
```markdown
> "原文英文引文"[^N]
> （中文译文）
```
