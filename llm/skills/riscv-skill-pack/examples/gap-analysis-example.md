# Gap Analysis Example

> 本文件演示 `gap-analysis` 子文件的典型用法，包含 Part 1（Gap 文档）
> 与 Part 2（评审 slides）两个阶段。所有内容均为**模拟**形态，用于
> 展示文档结构、source-tag 规范与 9 节模板，不代表任何真实 feature
> 的最终结论。

> **顺序约束**：Part 2（slides）必须以已完成的 Part 1 文档作为输入。
> 禁止在 slides 中重新推导证据——slides 只能引用 Part 1 的 source tag。

---

## 示例 1: Gap 文档生成（Part 1）

### 查询输入

```text
/skill gap-analysis
基于 MEMBER-00-GP 的 MMIO Outstanding 需求（REQ-MMIO-XXX），对比
RISC-V Privileged spec v1.12 与 Arm Neoverse V2，生成 F{N} gap
分析文档（9 节结构），并在 §4 执行 XiangShan 仓库复核。
```

### 期望输出形态

skill 应生成一个**9 节结构的 markdown 文档**，保存到交付目录：

**输出文件**：`<delivery-dir>/04_专题/_internal_Gap_Analysis_F{N}_MMIO_Outstanding.md`

**文档骨架**（模拟）：


## 示例 2: Gap 幻灯片生成（Part 2）

### 查询输入

```text
/skill gap-analysis
为上一步生成的 F{N} MMIO Outstanding gap 文档产出评审 slides，
严格 3 slides/feature，所有证据只能引用 Part 1 的 source tag，
不得重新推导。
```

### 期望输出形态

skill 应生成**每 feature 严格 3 slides** 的 PPTX：

**输出文件**：`<delivery-dir>/04_专题/_internal_Gap_Analysis_F{N}_MMIO_Outstanding_slides.pptx`

**Slide 结构**（模拟）：

```text
Slide 1 — Problem Statement
├─ Feature: MMIO Outstanding
├─ Requirement: REQ-MMIO-001 (P0, 8-deep) [T1-VERIFIED, §2.1]
├─ Gap: Privileged spec v1.12 §3.6 未定义 outstanding 上限
└─ Impact: R1 依赖自研扩展，V3R3 上游化风险

Slide 2 — Evidence & Cross-Check
├─ Tier-1 对比表（RISC-V / Arm Neoverse V2）[§3]
├─ XiangShan 复核: 8-deep, 硬编码 [§4, Layer 5]
└─ Source tag 分布: T1=92% [§1]

Slide 3 — Mitigation & Decision
├─ R1: 自研扩展 (现状)
├─ R2: 推动 RVI TG 立项 [§7]
├─ R3: 上游化 (目标)
└─ Open Question: Q1 [§8] — 待评审决议
```

**强制约束**（skill 自检）：
- [ ] 每个 feature **严格** 3 slides（不多不少）
- [ ] 所有引用必须指向 Part 1 文档的节号（§x.y），禁止引入新证据
- [ ] Source tag 必须与 Part 1 一致，不得新增 `[UNVERIFIED]` 条目
- [ ] PPTX 文件名同样以 `_internal_` 前缀标记
- [ ] 若 Part 1 文档不存在，skill 应**拒绝执行**并提示先运行 Part 1

> **顺序约束再强调**：Part 2 是 Part 1 的"投影"，不是重新分析。
> 如果 skill 试图在 slides 中引用 Part 1 之外的来源，视为流程违规。

## 跨子文件流程（推荐顺序）

完整的 gap 分析通常需要先建立证据基线：

```text
1. /skill spec-learning        → 用 tier-1 来源确认 feature 定义
2. /skill requirement-analysis → 找到匹配的 P0/P1 需求行
3. /skill gap-analysis Part 1  → 产出 9 节 gap 文档
4. /skill gap-analysis Part 2  → 基于 Part 1 产出 3 slides/feature
```

始终从 `spec-learning` 开始，确保 Part 1 的 source tag 基线为
Layer 3 (Ratified)。

## 占位符约定

| 占位符 | 代表 | 替换为 |
|--------|------|--------|
| `MEMBER-00-GP` | 你的旗舰产品线 | 实际产品代号 |
| `REQ-MMIO-XXX` | 需求编号示例 | 你的需求编号 |
| `<delivery-dir>` | 项目交付目录 | `your-project/04_专题/` |

| `V2R2` / `V3R3` / `V4R4` | 版本代际 | 你的版本规划 |
